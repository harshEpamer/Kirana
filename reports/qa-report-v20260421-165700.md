## Failures and Blockers

### [F-01] SQLite StaticPool missing in all conftest.py files — 42 tests fail
- **Severity:** Critical
- **Point of failure:** All conftest.py files in auth-service, catalog-service, inventory-service, sales-service
- **Probable reason:** `create_engine("sqlite:///:memory:", connect_args={"check_same_thread": False})` without `poolclass=StaticPool`. SQLite `:memory:` databases are connection-scoped — a new session connection opens a completely new, empty database. `Base.metadata.create_all` writes tables to one connection; the route handlers open new connections through `SessionLocal()` that see no tables.
- **Potential fix:** Add `poolclass=StaticPool` from `sqlalchemy.pool` to every conftest `create_engine` call:
  ```python
  from sqlalchemy.pool import StaticPool
  _TEST_ENGINE = create_engine(
      "sqlite:///:memory:",
      connect_args={"check_same_thread": False},
      poolclass=StaticPool,
  )
  ```

### [F-02] GET /auth/me is not implemented — returns HTTP 501
- **Severity:** Critical
- **Point of failure:** `backend/auth-service/routers/auth.py` line 64 — `raise HTTPException(status_code=501, detail="Wire up token dependency — see auth_utils pattern")`
- **Probable reason:** Endpoint was scaffolded but token-decoding dependency was never wired in.
- **Potential fix:** Add `get_current_user_id` dependency (as defined in conventions.instructions.md) and fetch user from DB by decoded `sub` claim.

### [F-03] No purchase_history records created on checkout
- **Severity:** High
- **Point of failure:** `backend/order-service/routers/orders.py` — POST `/orders/` creates `Sale` and `SaleItem` but never inserts into `purchase_history`.
- **Probable reason:** Step 7 from the Requirements.md agent prompt ("Insert into purchase_history") was omitted.
- **Potential fix:** After `db.flush()` (after sale is created), insert `PurchaseHistory(user_id=order_in.user_id, sale_id=sale.id)` and add it to the session before `db.commit()`.

### [F-04] No stock_adjustments records created on checkout
- **Severity:** High
- **Point of failure:** `backend/order-service/routers/orders.py` — stock is decremented via `product.stock_qty -= qty` but no `StockAdjustment` ORM object is inserted. The `StockAdjustment` model is not even imported or defined in order-service's `models.py`.
- **Probable reason:** The stock_adjustments requirement (step 6 in the agent prompt) was not implemented.
- **Potential fix:** Add `StockAdjustment` model to order-service's models.py and insert `StockAdjustment(product_id=product.id, adjustment_type="sale_deduct", quantity=qty)` for each item in the loop.

### [F-05] Alert-service check endpoint missing — inter-service call silently skipped
- **Severity:** High
- **Point of failure:** Requirements specify `GET /api/alerts/check/{product_id}` to be called per product after checkout. The alert-service only exposes `GET /alerts/low-stock` — no `check/{product_id}` endpoint exists. Order-service makes no inter-service call at all.
- **Probable reason:** The inter-service alert trigger specified in requirements was never implemented in either service.
- **Potential fix:** Add `GET /alerts/check/{product_id}` endpoint to alert-service; add async httpx call from order-service post-checkout.

### [F-06] Checkout sends no user_id — all orders are anonymous
- **Severity:** High
- **Point of failure:** `frontend/src/pages/CartPage.jsx` — `checkout({items: [...], coupon_code: ...})` passes no `user_id`. `order-service/schemas.py` makes `user_id` optional and defaults to `None`. All sales are stored with `user_id = NULL`.
- **Probable reason:** AuthContext does not expose a `user.id` field after login (token JWT payload decoded into `user` state is not populated post-login — see F-07).
- **Potential fix:** Decode JWT sub claim in AuthContext on login to set `user.id`; pass `user_id` in checkout call.

### [F-07] AuthContext.user is always null after login — no token decode on login
- **Severity:** High
- **Point of failure:** `frontend/src/context/AuthContext.jsx` — `login()` stores the token but never parses it. `user` state remains `null` indefinitely. Frontend UI cannot read `user.id`, `user.name`, or `user.role`.
- **Probable reason:** The `useEffect` that would decode `token` and restore user state was not implemented — only token persistence exists.
- **Potential fix:** After a successful login, decode the JWT (e.g., using `atob` on the payload section) or call `GET /auth/me` to fetch user data and `setUser(data)`.

### [F-08] No JWT authentication on any admin/mutation endpoints
- **Severity:** High
- **Point of failure:** All routes in inventory-service, order-service, coupon-service, sales-service, customer-service, alert-service have no Bearer token verification. Anyone can `DELETE /inventory/products/{id}`, `GET /customers/`, etc. without a token.
- **Probable reason:** The `get_current_user_id` dependency from conventions.instructions.md was not applied to any route.
- **Potential fix:** Apply the JWT dependency to all protected routes using `Depends(get_current_user_id)`.

### [F-09] No admin role in User model or JWT — admin guard impossible to enforce
- **Severity:** High
- **Point of failure:** `backend/auth-service/models.py` — `User` model has no `role` column. JWT payload encodes only `{sub, phone}`. Frontend `Navbar.jsx` checks `user.role === 'admin'` which is always `undefined` (and `user` is always `null` — see F-07).
- **Probable reason:** Role-based access was mentioned in requirements but never implemented in the schema or JWT issuance.
- **Potential fix:** Add `role TEXT DEFAULT 'customer'` column to `users` table (and update schema.sql); seed admin user with `role='admin'`; include `role` in JWT payload; decode and expose `user.role` in AuthContext.

### [F-10] No AuthGuard / protected routes in App.jsx
- **Severity:** High
- **Point of failure:** `frontend/src/App.jsx` — all routes including `/admin` and `/inventory` are publicly accessible without any authentication guard.
- **Probable reason:** The `AuthGuard` component specified in Requirements.md was never implemented.
- **Potential fix:** Create an `AuthGuard` component that reads `token` from `AuthContext` and redirects to `/login` if absent. Wrap all protected routes in it.

### [F-11] Pydantic v1-style `class Config` in all schemas — deprecation warnings
- **Severity:** Low
- **Point of failure:** `schemas.py` in auth-service, catalog-service, inventory-service, sales-service — all use `class Config: from_attributes = True` (Pydantic v1 style).
- **Probable reason:** Code generation used v1 patterns on Pydantic v2.
- **Potential fix:** Replace `class Config: from_attributes = True` with `model_config = ConfigDict(from_attributes=True)` using `from pydantic import ConfigDict`.

## Code Quality Findings

### Code Smells

- **sales-service/routers/sales.py — N+1 queries in list_sales and get_sale:** `list_sales()` fetches all sales then issues a separate `db.query(SaleItem)` per sale row. For 12 seeded sales, this is 13 queries. Should use a join or `joinedload`.
- **customer-service/routers/customers.py — N+1 in get_purchase_history:** For each `PurchaseHistory` record, a separate `db.query(Sale)` is issued. Should join `PurchaseHistory` → `Sale` in one query.
- **order-service/routers/orders.py — Redundant coupon lookup:** Coupons are queried from the `order-service`'s own DB. But the coupon-service is supposed to be the authority. The order-service re-implements coupon discount logic (`_apply_coupon`) that already exists in the coupon-service, creating duplication and a divergence risk.

### Duplication

- `_apply_coupon` in `order-service/routers/orders.py` duplicates the discount calculation already in `coupon-service/routers/coupons.py`. Two independent implementations of the same business rule.
- Stock-deduction logic duplicated between `sales-service` and `order-service` — both decrement `product.stock_qty` directly. If both are called in sequence (e.g., via `salesApi.createSale` from admin + the checkout flow), double-deduction occurs.

### Bad Practices

- **Hardcoded test expectation `test_get_me_returns_501`:** A test that asserts a 501 response treats an unimplemented endpoint as a passing state. This will mask when the endpoint is finally broken in a different way.
- **`user_id` is `Optional` in `OrderCreate`:** Orders can be placed without a user identity, making purchase history and multi-user analytics impossible.
- **`datetime.utcnow()` used in Python 3.12** — `datetime.utcnow()` is deprecated since Python 3.12; should use `datetime.now(timezone.utc)`.
- **`created_at` set in service layer** — `created_at=datetime.utcnow().isoformat()` rather than relying on the DB default `DEFAULT (datetime('now'))`. Timezone handling is inconsistent (service uses UTC string, DB uses SQLite local time).

### Vulnerabilities

- **No rate limiting on POST /auth/login or POST /auth/register:** Brute-force attacks and phone enumeration are possible with no throttling.
- **JWT SECRET_KEY fallback is a weak default:** `"kirana-secret-key-change-in-production"` will be used in any environment where `JWT_SECRET` environment variable is not set. In a deployed context this is a security leak. Should fail-fast if `JWT_SECRET` is not set via an environment variable.
- **All admin endpoints unauthenticated (see F-08):** Any client can mutate inventory, delete products, deactivate coupons, or read full customer lists without credentials.
- **Phone number used as credential without format validation:** No regex or length validation on phone numbers in `UserCreate`. Risk of phone enumeration via timing side-channel on login.

### Efficiency Concerns

- **N+1 DB queries in `list_sales` (sales-service):** For 37 seeded sale_items across 12 sales, this issues 13 DB queries per page load. Should use `joinedload(Sale.items)` or a single query with a join.
- **N+1 DB queries in `get_purchase_history` (customer-service):** Each purchase history entry triggers a separate `Sale` lookup.
- **All catalog queries are unauthenticated and uncached:** High-frequency read endpoints (`GET /catalog/products`) re-query the DB on every request with no caching layer. Acceptable for hackathon scope but noted.
- **CatalogPage re-fetches all products on every keystroke** (search `onChange` triggers a new `useEffect` with the full search query to the server). No debounce applied — risk of API churn on fast typing.

## Recommendation

### Gate decision: **NOT READY for human testing**

### Rationale

The automated test suite is systematically broken — 42 of 52 tests fail due to a shared SQLite in-memory pool misconfiguration that causes all database-touching tests to crash before reaching any assertion. This means there is **zero verified functional coverage** from automated tests.

Beyond the test infrastructure, code inspection reveals multiple critical and high-severity gaps that would block a complete demo flow:

1. The checkout flow does not record purchase history or stock adjustment logs.
2. GET /auth/me is unimplemented — the admin dashboard and user-specific features that depend on knowing who is logged in cannot work.
3. No admin role infrastructure exists in the database, JWT, or frontend routing — admin-only pages are open to all and the Navbar admin detection always fails.
4. No protected routes — any unauthenticated request can mutate inventory, create sales, or deactivate coupons.
5. All orders are created with `user_id = null` because the frontend never decodes the JWT to extract the user's ID.
